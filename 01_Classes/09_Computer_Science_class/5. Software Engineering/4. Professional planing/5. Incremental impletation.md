## 📝 Notes
*   **1. Micro‑verification loop 🌀**

Work in this rhythm:

1. Write a tiny piece (one function or one small change).
2. Add a **quick print check** with simple, clear inputs.
3. Run it.
4. If it’s wrong, fix _this_ slice before adding more.

You did this with:

- `filter_evens` → `print(filter_evens([1,2,3,4,5,6]))`
- `average_evens` → `print(average_evens([1,2,3,4,5,6]))`
- `average_list` → (we discussed checks like `[2,4,6]` and `[]`)

Mantra: **“Make it run, then make it right, then make it clean.”**

---

**2. Smallest testable unit 🔍**

Instead of thinking “build the whole feature,” think:

> “What is the smallest piece I can build and test _right now_ that makes real progress?”

In our example:

- Smallest unit 1: `filter_evens(numbers)`
- Smallest unit 2: `average_list(numbers)`
- Integration: `average_evens(numbers)` = glue of the two

For `max_even(numbers)`, you correctly picked:

- First: reuse `filter_evens(numbers)` to narrow data
- Then: add “take `max` or return `None` if empty”

---

**3. Single responsibility per function 🎯**

Each function should answer **one clear question**:

- `filter_evens` → “Which numbers from this list are even?”
- `average_list` → “What is the average of this list (or `None` if it’s empty)?”
- `average_evens` → “Using those helpers, what is the average of the even numbers in this list?”

If you catch a function doing _two_ different jobs, that’s a signal:

- Extract one job into its own helper.
- Wire the old function to call the new one (like we did with `average_list`).

---

**4. Reuse > Recompute 🧠**

When adding new behavior:

- First ask: **“What do I already have that gets me closer?”**
    - Example: for `max_even`, reuse `filter_evens` instead of re‑implementing filtering or misusing `average_list`.
