# 2.Knowing when done is done

## 📌 Overview
*   **Course:** SE
*   **Lecture Date:** 2026-05-10 
*   **Instructor:** maestro

## 🎯 Learning Objectives
*   **1. Boundary conditions & edge cases**

- A **boundary value** is one of the _limits_ of what’s allowed.
    
    - Example: rule = “age 13–120 inclusive”
        - Boundaries: 13 and 120
        - Just outside: 12 and 121
- **Edge case (in this lesson’s sense)** = a _barely valid_ input, right on the allowed edge.
    
    - 13 → barely valid (low edge)
    - 120 → barely valid (high edge)
- **Plain invalid** = just wrong, not “interesting” as a boundary.
    
    - 10, 0, 200, -5 for the rule 13–120 → all invalid, but not boundaries.

Think:

> Edge case = “last number still allowed” or “first number still allowed”.

**2. Barely valid vs. invalid — table mental model**

For rule: “username must be 3–10 characters, letters only”:

|Input|Category|Why|
|---|---|---|
|`ab`|Plain invalid|2 chars, < 3|
|`abc`|Edge case (valid)|Exactly 3 (low boundary)|
|`abcdefghij`|Edge case (valid)|Exactly 10 (high boundary)|
|`abcdefghijk`|Plain invalid|11 chars, > 10|

Only **abc** and **abcdefghij** belong in the _edge-case list_.

**3. Acceptance criteria – core pattern**

Good acceptance criteria are:

- Specific
- Binary (pass/fail)
- Testable

Simple pattern:

> **Given** <context>,  
> **when** <user action / input>,  
> **then** <clear expected result>.

Examples (age 13–120):

- Valid edge (13):
    
    > Given a signup form that only accepts ages from 13 to 120,  
    > when the user enters age 13,  
    > then the form accepts the age and allows the signup to continue.
    
- Just-below boundary invalid (12):
    
    > Given a signup form that only accepts ages from 13 to 120,  
    > when the user enters age 12,  
    > then the form rejects the age as invalid and does not allow signup to continue.
    

Bad acceptance criteria examples (what to avoid):

- “The system should work correctly for most ages.” → too vague.
- “The system handles it somehow.” → no clear expected result.

**4. From acceptance criteria → “Definition of done”**

For a rule like:

> “Username must be 3–10 letters.”

You might define these acceptance criteria:

- 3-char username is accepted.
- 10-char username is accepted.
- 2-char username is rejected.
- 11-char username is rejected.

Each one becomes a **checkbox** on your “done” list.  
When every checkbox passes in testing, this part of the feature is **done**.

**5. How to think fast on future tasks**

When you see a rule/spec:

1. **Find boundaries**
    
    - Minimum allowed? Maximum allowed?
    - “At least X”? → boundaries: X and X−1
    - “Up to Y inclusive”? → boundaries: Y and Y+1
2. **List 2–4 key test inputs**
    
    - Low edge and just-below
    - High edge and just-above
3. **Turn each into Given/When/Then**
    
    - Given: page/state/form
    - When: exact input (number, length, etc.)
    - Then: accept + proceed OR reject + block

That’s it. That’s the whole practical skill.
