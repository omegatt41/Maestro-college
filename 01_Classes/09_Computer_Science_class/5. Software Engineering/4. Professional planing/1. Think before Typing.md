## 🎯 Learning Objectives
*   feature specialization
- **Inputs:** one `username` string
- **Outputs:** `True` or `False` (username is valid or not)
- **Constraints:** specific rules the username must follow (like length 3–12, only letters/numbers, etc.)
- **Edge cases:** inputs that sit on or outside those rules (too short, too long, weird characters, empty, etc.)
## 📝 Notes
* **Feature spec template**

Use this 4‑box layout every time:

|Section|What you write there|
|---|---|
|**Inputs**|Exact data the feature receives (names + types). One line per input.|
|**Outputs**|Exact data it returns/produces (names + types, or True/False, message, etc.).|
|**Constraints**|Short, testable rules that must always be true. One simple sentence per rule.|
|**Edge cases**|Concrete example inputs that sit on/over the boundaries of your rules.|

**Style notes**

- Inputs/outputs:
    
    - Use phrases like: `one username string`, `one password string`, `integer age`, etc.
    - Avoid vague stuff like “data” or “info”.
- Constraints:
    
    - Each rule = **1 clear sentence**, e.g.
        - “Username length must be between 3 and 12 characters.”
        - “Password must contain at least one uppercase and one lowercase letter.”
- Edge cases:
    
    - Always **specific examples**, e.g. `"Ad!1"`, `"No1"`, `"Long-pas$w0rd34!"`
    - Mark them as “barely valid” or “barely invalid”.
