## 📝 Notes & Edge Cases
- **Key Concepts:** 
	- **1. Core idea (mental model)**

	- Variable = **label**
	- Object = **locker / box**
		- Two labels on **one locker** → same object (aliasing)
		- Two separate lockers with same contents → different objects, equal-looking values

	-**2. Aliasing (same object)**

Python

```python
class Locker:
    pass

a = Locker()
b = a      # aliasing: a and b point to SAME object

a.code = 1234
print(b.code)  # 1234
print(a is b)  # True
```

- Changing `a.code` also changes what you see from `b.code`
- Any mutation through `a` or `b` affects the **same underlying object**

**3. Separate objects, same values**

Python

```python
c = Locker()
d = Locker()

c.code = 1234
d.code = 1234

print(c is d)  # False  (different objects)
print(c.code)  # 1234
print(d.code)  # 1234
```


- Same values in attributes
- Different objects → changing `c.code` does **not** change `d.code`

**4. `is` vs `==` for custom objects**

- `is` → checks **identity** (same object in memory?)
- `==` → normally checks **value equality**, but:
    - For a simple class like `class Locker: pass`, `==` defaults to acting like identity
    - So, for `Locker` without any extra logic:
        - `x is y` and `x == y` are both `False` when `x` and `y` are different objects

Example:

Python

```python
x = Locker()
y = Locker()

x.code = 5555
y.code = 5555

print(x is y)  # False
print(x == y)  # False  (same as `is` for plain custom objects)
```

**5. Why identity vs equality matters**

- **Aliasing (same object)** is risky:
    - Change via one name → surprises code using the other name
- **Different objects, same values**:
    - Changes stay local to that one object
- So you use `is` when you care about **shared state / same object**, especially with custom objects.
