# definitions:
**Object:** a bundle of related data and behavior kept together
		-Example: think of a **dog** in a simple game 🐶:
			- state(Data): name, age, breed
			- Behavior: bark, eat, sleep
			- (Instead of having separate variables and separate functions all over the place, OOP says: “Put the dog’s data _and_ what it can do together in one object.”)
			- An object contains two parts data and behavoir(ex. ‘dog’)
				- state(data) =  breed, name, age,
				- behavoir = bark, run,eat sleep
				- State holds the object's data • Behavior defines what it can do •Together they form an **Object**
# History (how we got to oop):
	1. **1950s–60s: Procedural style**
    
    - Programs were mostly long step-by-step procedures: “do this, then this, then this.”
    - Worked fine for small programs.
2. **1970s–80s: Software got huge 😵**
    
    - Bigger systems (GUIs, simulations, business software) became hard to manage as just thousands of steps.
    - Bugs were hard to track because everything was tangled.
3. **OOP appears**
    
    - Languages like **Smalltalk** and later **C++** said:
        
        > “Let’s model programs as _objects_ that represent real concepts (like `Dog`, `Car`, `BankAccount`) instead of just raw steps.”
        
    - This made it easier to think about, grow, and maintain big systems.

Python came later (1990s) and **built in OOP ideas from the start**.
